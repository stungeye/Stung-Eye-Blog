# Eleventy System Guide

A detailed walkthrough of how this Eleventy site works — configuration, data flow, template rendering, collections, pagination, and important edge cases.

## Eleventy 3.x and ESM

This project uses Eleventy 3.x, which supports native ES modules. The `package.json` has `"type": "module"`, so all `.js` files use `import`/`export` syntax instead of `require`/`module.exports`. The config file uses `export default function` rather than the 2.x-era `module.exports`.

```js
// eleventy.config.js (3.x style)
export default function (eleventyConfig) {
  // ...config here...
}
```

If you're coming from 2.x, the main differences are:

- ESM imports throughout (including data files)
- Config file uses `export default` instead of `module.exports`
- The API is otherwise very similar

## Directory Structure

```
src/                        ← Eleventy input directory
  _data/config.js           ← Global data (available as `config` in templates)
  _includes/
    layouts/                ← Layout chain templates
      base.njk              ← HTML shell (head, nav, footer)
      day.njk               ← Single day page layout
      archive.njk           ← Paginated archive layout
      archive-index.njk     ← Year/month/archive landing layout
    partials/
      nav.njk               ← Nav bar (currently inlined in base.njk too)
      pagination.njk        ← Prev/next pagination controls
  pages/                    ← Page templates (homepage, archive, sitemap, etc.)
  posts/                    ← Day page content (generated by migration script)
    posts.json              ← Directory data file (assigns layout to all posts)
    2002/08/18/index.md
    2008/07/23/index.md
    ...
assets/                     ← Static assets (outside src/, copied as-is)
  css/main.css
_site/                      ← Build output (gitignored)
```

The returned config object tells Eleventy where things are:

```js
return {
  dir: {
    input: "src", // All source content
    includes: "_includes", // Relative to input — layouts and partials
    data: "_data", // Relative to input — global data files
    output: "_site", // Build output
  },
  markdownTemplateEngine: false, // Explained below
  htmlTemplateEngine: "njk",
  templateFormats: ["md", "njk"],
};
```

## Global Data

`src/_data/config.js` exports an object that becomes available as `config` in every template:

```js
export default {
  siteTitle: "Stung Eye",
  siteUrl: "https://www.stungeye.com",
  daysPerPage: 10,
};
```

The filename determines the variable name — `config.js` → `config`. Any `.js` or `.json` file in `_data/` becomes a global data variable. If you added `src/_data/foo.json`, you'd access it as `foo` in templates.

## How Day Pages Are Built

### Content Files

Each day page lives at `src/posts/YYYY/MM/DD/index.md` with YAML frontmatter:

```yaml
---
date: 2008-07-23 18:46:42
title: Discovered on July 23, 2008
permalink: /archive/by_date/2008/07/23/
---
Content body (markdown and inline HTML)...
```

The three frontmatter fields:

- **`date`**: The intended publish/archive datetime. Migrated bare datetimes are
  legacy site-local wall time, effectively `America/Winnipeg`, not UTC
  instants. Eleventy/js-yaml currently parses these bare values into JavaScript
  `Date` objects as if they were UTC. The planned repair is to update migration
  so generated frontmatter datetimes include explicit Winnipeg offsets, allowing
  Eleventy to ingest them normally as real instants.
- **`title`**: Used in the page `<title>`, the `<h1>`, and when other pages link to this day.
- **`permalink`**: Overrides Eleventy's default URL derivation. Without this explicit permalink, Eleventy would derive URLs from the file path — which would give `/posts/2008/07/23/` instead of `/archive/by_date/2008/07/23/`. The trailing slash causes Eleventy to generate an `index.html` file at that path.

### Directory Data File

`src/posts/posts.json` assigns a layout to every file under `src/posts/`:

```json
{
  "layout": "layouts/day.njk"
}
```

This is an Eleventy concept called a **directory data file**. Instead of putting `layout: layouts/day.njk` in every post's frontmatter, you set it once in a `.json` (or `.js`) file named after the directory. It applies to all templates in that directory and its subdirectories.

If you wanted to add another default frontmatter value to every post — say, `tags: post` — you'd add it here.

### Layout Chain

When Eleventy processes `src/posts/2008/07/23/index.md`, the rendering chain is:

```
index.md → day.njk → base.njk → final HTML
```

1. **index.md**: Markdown is converted to HTML. The result is the `content` variable.
2. **day.njk**: Wraps `content` in an `<article>` with the title and date. Its own output becomes the new `content`.
3. **base.njk**: Wraps everything in the HTML shell — `<head>`, `<nav>`, `<main>`, `<footer>`.

`day.njk` declares its parent layout in its own frontmatter:

```njk
---
layout: layouts/base.njk
---
<article class="day-page">
  <header class="day-header">
    <h1>{{ title }}</h1>
    <time datetime="{{ page.date | isoDate }}">{{ page.date | readableDate }}</time>
  </header>
  <div class="day-content">
    {{ content | safe }}
  </div>
</article>
```

The `| safe` filter is critical — without it, Nunjucks would HTML-escape the rendered content, turning all your `<p>` tags into visible `&lt;p&gt;` text.

### The `markdownTemplateEngine: false` Setting

This is the most important edge case in the whole system.

By default, Eleventy pre-processes markdown files through a template engine (Nunjucks) _before_ converting the markdown to HTML. This means if your markdown content contains `{{ anything }}` or `{% anything %}`, Nunjucks will try to evaluate it as a template expression.

The legacy content includes literal `{{ }}` characters in several posts (code snippets, old formatting artifacts). With the default setting, Eleventy would crash or produce garbled output on those posts.

Setting `markdownTemplateEngine: false` tells Eleventy: **do not run Nunjucks on markdown files — just convert the markdown to HTML and pass it to the layout.**

The tradeoff: you cannot use Nunjucks expressions inside markdown files. No `{{ title }}` or `{% include %}` in `.md` content. This is fine because:

- Post frontmatter (YAML) still works normally — Eleventy parses it regardless of template engine settings.
- All template logic lives in `.njk` layout and page files, which are unaffected by this setting.
- Post content is pre-rendered HTML/markdown that doesn't need template processing.

If you ever needed Nunjucks in a new markdown post, you could set `templateEngineOverride: njk,md` in that file's frontmatter — but be careful, because the build will fail if any `{{ }}` appears in that file's content without being a valid Nunjucks expression.

## Collections

Collections are how Eleventy makes groups of pages available to templates. This site defines three custom collections in `eleventy.config.js`.

### `collections.days`

```js
eleventyConfig.addCollection("days", (collectionApi) => {
  return collectionApi
    .getFilteredByGlob("src/posts/**/index.md")
    .sort((a, b) => b.date - a.date);
});
```

All 970 day pages, sorted newest first. Each element is an Eleventy page object with properties like:

- `.date` — JavaScript `Date` from frontmatter. After the date repair, migrated
  datetimes should include explicit offsets so this value is a real instant.
- `.url` — the permalink (`/archive/by_date/2008/07/23/`)
- `.data` — frontmatter values (`.data.title`, `.data.permalink`, etc.)
- `.content` — rendered HTML content of the page

This collection drives the homepage pagination and the sitemap.

### `collections.daysByYear`

```js
eleventyConfig.addCollection("daysByYear", (collectionApi) => {
  // ... groups days and returns:
  return Object.entries(byYear).sort((a, b) => b[0].localeCompare(a[0]));
});
```

Returns an array of `[year, days[]]` pairs sorted newest year first:

```js
[
  ["2025", [day, day, ...]],
  ["2024", [day, day, ...]],
  // ...
  ["2002", [day, day, ...]]
]
```

Used by `year.njk` (paginates over each entry to create one page per year) and `archive.njk` (iterates to build the archive landing page).

**Why arrays, not Maps or objects?** Eleventy's pagination system requires array data. Maps and plain objects don't work with `pagination.data`. Returning `Object.entries()` gives an array of tuples that pagination can iterate over.

### `collections.daysByMonth`

Same pattern as `daysByYear`, but keyed by `"YYYY-MM"` strings:

```js
[
  ["2025-03", [day, day, ...]],
  ["2025-02", [day, day, ...]],
  // ...
]
```

Used by `month.njk` to generate one page per month.

## Pagination

Eleventy pagination lets one template generate multiple pages from a data array. This site uses it in three places.

### Homepage (`src/pages/index.njk`)

```yaml
pagination:
  data: collections.days
  size: 10
  alias: days
  reverse: false
permalink: "{% if pagination.pageNumber == 0 %}/{% else %}/page/{{ pagination.pageNumber + 1 }}/{% endif %}"
```

- **`data`**: The collection to paginate over.
- **`size: 10`**: 10 days per page. (Matches `config.daysPerPage`, but pagination size must be a literal in frontmatter — it can't reference a variable.)
- **`alias: days`**: Inside the template, the current page's chunk of 10 days is available as `days`.
- **`reverse: false`**: The collection is already sorted newest-first, so no reversal needed.
- **`permalink`**: Dynamic — page 0 gets `/`, subsequent pages get `/page/2/`, `/page/3/`, etc. The `pagination.pageNumber` is 0-indexed.

The template loops over `days` and renders each day's full content inline, then includes the pagination partial for prev/next links.

> **If you change `daysPerPage` in config.js**, you also need to update the `size: 10` value in `index.njk` frontmatter.

### Year Pages (`src/pages/year.njk`)

```yaml
pagination:
  data: collections.daysByYear
  size: 1
  alias: yearEntry
permalink: "/archive/by_date/{{ yearEntry[0] }}/"
```

With `size: 1`, each entry in the array gets its own page. `yearEntry` is a `[year, days[]]` tuple, so `yearEntry[0]` is the year string and `yearEntry[1]` is the array of day pages for that year.

The template uses `eleventyComputed` for the title:

```yaml
eleventyComputed:
  title: "Archive: {{ yearEntry[0] }}"
```

`eleventyComputed` lets you set frontmatter values that depend on other data — here, the page title depends on which year this page is for. These values are computed at render time, not parse time.

### Month Pages (`src/pages/month.njk`)

Same pattern with `collections.daysByMonth` and `size: 1`. Each `monthEntry` is a `["YYYY-MM", days[]]` tuple. The permalink uses a `replace` filter to convert `"2008-07"` to `"2008/07"` for the URL path.

Month pages render full day content inline (like the homepage), with a back link to the year page.

## Filters

Custom Nunjucks filters defined in `eleventy.config.js`. Filters transform values in templates using the `|` pipe syntax: `{{ value | filterName }}`.

| Filter          | Input              | Output                       | Used In                         |
| --------------- | ------------------ | ---------------------------- | ------------------------------- |
| `readableDate`  | date/page value    | `"July 23, 2008"`            | Day pages, day blocks           |
| `shortDate`     | date/page value    | `"Jul 23, 2008"`             | (Available, not currently used) |
| `isoDate`       | date/page value    | site-time ISO datetime       | `<time datetime>`, sitemap      |
| `yearFromDate`  | date/page value    | `"2008"`                     | (Available for templates)       |
| `monthFromDate` | date/page value    | `"07"` (zero-padded)         | Archive page month extraction   |
| `monthName`     | `"07"` or `7`      | `"July"`                     | Archive/year/month pages        |
| `split`         | `"2008-07"`, `"-"` | `["2008", "07"]`             | Month page title computation    |
| `absoluteUrl`   | `"/path/"`, base   | Full URL                     | Sitemap                         |

Date handling should be unified around the configured site timezone,
`America/Winnipeg`. The important trap: formatting Eleventy's already-parsed
`Date` in Winnipeg is still wrong for migrated bare datetimes, because those
values have already been misread as UTC. The repair direction in
`docs/issues-and-fixes.md` issue #5 is to make migrated frontmatter explicit
with Winnipeg offsets. After that, Eleventy can ingest `date` normally, while
archive grouping and display should use the configured site timezone rather than
UTC or the build machine's local timezone.

## Passthrough Copy

Passthrough copy tells Eleventy to copy files into the output unchanged, without processing them as templates.

### Static Assets

```js
eleventyConfig.addPassthroughCopy("assets");
```

Copies the entire `assets/` directory to `_site/assets/`. Since `assets/` is outside `src/`, the path is relative to the project root. The CSS is then available at `/assets/css/main.css`.

### Co-located Images

```js
eleventyConfig.addPassthroughCopy(
  { "src/posts/": "archive/by_date/" },
  {
    filter: [
      "**/*.jpg",
      "**/*.jpeg",
      "**/*.png",
      "**/*.gif",
      "**/*.webp",
      "**/*.svg",
      "**/*.bmp",
    ],
  },
);
```

This is the trickiest passthrough rule. It does two things:

1. **Path remapping**: The object syntax `{ "src/posts/": "archive/by_date/" }` means "copy from `src/posts/` but place the output under `archive/by_date/`". An image at `src/posts/2008/07/23/photo.jpg` lands at `_site/archive/by_date/2008/07/23/photo.jpg`.

2. **Filtering**: Only image files are copied — the `index.md` files are left for Eleventy to process as templates. Without the filter, Eleventy would also copy the markdown files as raw text.

This remapping is necessary because the content lives under `src/posts/` (Eleventy convention), but URLs reference `/archive/by_date/` (site convention for backward compatibility). The markdown files handle this automatically via their `permalink` frontmatter, but images need the passthrough copy to put them in the right place.

## Page Templates

### Sitemap (`src/pages/sitemap.njk`)

```yaml
permalink: /sitemap.xml
eleventyExcludeFromCollections: true
```

`eleventyExcludeFromCollections: true` prevents the sitemap itself from appearing in any collection (and thus in the sitemap's own URL list, or in pagination data).

The sitemap specifically calculates the number of paginated homepage pages using `collections.days | length / config.daysPerPage` and generates URLs for pages 2 through N. It also lists every day page, year page, month page, and the archive landing.

### robots.txt (`src/pages/robots.njk`)

Same pattern — `.njk` template with a custom permalink and collection exclusion. The `.njk` extension lets Eleventy process it as a template (to inject `config.siteUrl`), even though the output is a plain text file.

## Canonical URLs and SEO

Every page has a `<link rel="canonical">` tag in `base.njk`:

```html
<link rel="canonical" href="{{ config.siteUrl }}{{ page.url }}" />
```

`page.url` is a built-in Eleventy variable containing the page's URL path (e.g., `/archive/by_date/2008/07/23/`). Combined with `config.siteUrl`, this produces the full canonical URL.

All pages are self-canonical — day pages point to themselves, homepage points to `/`, page 2 points to `/page/2/`. Paginated pages do **not** canonical back to page 1.

## How Content Flows Through the System

To make the full pipeline concrete, here's what happens for one day page during `npm run build`:

1. Eleventy discovers `src/posts/2008/07/23/index.md`.
2. It reads `src/posts/posts.json` and inherits `layout: layouts/day.njk`.
3. YAML frontmatter is parsed: `date`, `title`, `permalink`.
4. Because `markdownTemplateEngine: false`, the body is **not** run through Nunjucks. It goes straight to the markdown processor.
5. Markdown is converted to HTML → this becomes `content`.
6. `day.njk` renders, wrapping `content` in article markup, using `title` and `page.date` for the header. Its output becomes the new `content`.
7. `base.njk` renders, wrapping everything in the HTML shell.
8. The final HTML is written to `_site/archive/by_date/2008/07/23/index.html` (per the `permalink`).
9. Any images in `src/posts/2008/07/23/` matching the passthrough filter are copied to `_site/archive/by_date/2008/07/23/`.

For the homepage, a similar flow happens — but `index.njk` paginates over `collections.days`, accessing each day's already-rendered `.content` and outputting it inline.

## Adding New Content

Create a new day page at `src/posts/YYYY/MM/DD/index.md`:

```yaml
---
date: 2026-04-04 12:00:00
title: New Post Title
permalink: /archive/by_date/2026/04/04/
---
Write content here. Markdown works. Inline HTML works.

![Alt text](/archive/by_date/2026/04/04/photo.jpg)
```

Place images in the same directory. Reference them with absolute paths starting from `/archive/by_date/`.

The post will automatically:

- Use `layouts/day.njk` (from `posts.json`)
- Appear in `collections.days` (sorted by date)
- Show up on the homepage and in month/year archives
- Be included in the sitemap

Things to remember:

- You **cannot** use `{{ }}` or `{% %}` syntax in `.md` files (template engine is disabled).
- The `permalink` must be set explicitly — Eleventy won't derive the right URL from the file path alone.
- Image paths must be absolute, not relative, so they resolve correctly from any page (homepage, month page, etc.).
- The `date` field controls sort order and archive grouping. Migrated datetimes
  should include explicit Winnipeg offsets after issue #5 is repaired; keep the
  frontmatter date, folder path, and permalink calendar day aligned in
  `America/Winnipeg`.

## CSS

Hand-written CSS in `assets/css/main.css`. No framework, no preprocessor.

Key layout decisions:

- Content max-width: `48rem` (centered)
- Font: Georgia serif stack
- Responsive breakpoint at `600px`
- Day blocks on homepage/month pages are separated by `<hr class="day-separator">`
- Images have `max-width: 100%` for responsive sizing
